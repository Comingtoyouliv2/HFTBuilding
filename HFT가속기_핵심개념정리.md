# HFT 가속기 (FPGA) — 핵심 개념 정리

> MIT 6.111 기말 프로젝트 *"An HFT Accelerator"* (Endrias·Natnael·Tony, 2019)를 실제 FPGA 보드로 재현하기 위해 먼저 잡고 가야 할 배경 지식 모음입니다.
> 목표: 거래소 데이터 → 파싱 → 오더북 → 트레이딩 로직 → 주문 생성까지 전체 파이프라인을 하드웨어로 구현.

---

## 0. 한눈에 보는 전체 그림

```
[Laptop = 거래소 시뮬레이터]
        │  ITCH 프로토콜 메시지 (TCP/Ethernet)
        ▼
┌──────────────────────────────────────────────────────────┐
│  FPGA                                                      │
│                                                            │
│  Ethernet/TCP ── Parser ── Order Book ── Trading Logic     │
│   (MicroBlaze)   (ITCH)    (최우선호가)   (최소위험 포트폴리오) │
│                                │              │            │
│                                └── Risk 계산   └── 주문 생성  │
│                                        │          │         │
└────────────────────────────────────────┼──────────┼─────────┘
                              VGA/HEX 표시  │  주문 (OUCH/Ethernet)
                                            ▼
                                       [Laptop 표시]
```

**핵심 가치**: 지연(latency)이 낮을 뿐 아니라 **결정적(deterministic)** 이다. CPU는 메모리 계층 때문에 같은 작업도 실행 시간이 들쭉날쭉하지만(원본 측정: 800μs~1.6ms 변동), FPGA는 매번 같은 사이클 수로 끝난다. HFT에서는 이 예측 가능성이 절대적이다.

## 1. 도메인 용어 (금융)

| 용어 | 의미 |
|------|------|
| **Stock(종목)** | 사고팔 수 있는 가상의 대상 |
| **Order(주문)** | 종목을 사거나 팔겠다는 요청 |
| **Bid(매수호가)** | 특정 가격에 **사겠다**는 주문 |
| **Ask(매도호가)** | 특정 가격에 **팔겠다**는 주문 |
| **Exchange(거래소)** | 매수·매도를 매칭하고, 시장 업데이트를 네트워크로 뿌리는 제3자 |
| **Automated trading** | 거래소에 연결해 돈을 벌 목적으로 주문을 내는 컴퓨터 전략 |
| **HFT** | 자동매매의 일종. 시장 변화에 **초저지연으로** 반응해 주문을 내는 것이 목표 |

HFT 회사가 빠르게 해야 하는 3가지:
1. 거래소 데이터스트림을 **파싱**한다.
2. 시장 상태(각 종목의 최우선 호가)를 추적한다 = **오더북 구축**.
3. 그 정보를 바탕으로 **주문을 제출**한다.

상용 솔루션은 1·2단계가 각각 ~300ns 수준. 3단계는 전략 복잡도에 따라 다름. 이 프로젝트의 목표는 "제한된 자원으로 얼마나 근접하는가"이다.

## 2. ITCH 프로토콜 (파서가 다루는 데이터)

NASDAQ이 시장 데이터를 내보낼 때 쓰는 **바이너리 프로토콜**. 주문 추가/취소/체결 같은 시장의 시간적 변화를 메시지로 전달한다.

### 메시지 구조
- **빅엔디언(big-endian)** 바이트 스트림
- 메시지 = `[길이][메시지 타입][내용]`
- 메시지 타입이 1바이트로 구조화되어 있어서, **바이트 단위 FSM**으로 파싱하면 구현이 단순해진다 (원본의 핵심 설계 결정)

### Add Order 메시지 예시
| 길이(byte) | 필드 | 의미 |
|-----------|------|------|
| 1 | Message type | `'A' = 0x41` → 주문 추가 |
| 2 | Stock locate | 어떤 종목인지 |
| 2 | Tracking number | NASDAQ 내부 추적 번호 |
| 6 | Timestamp | 개장 후 경과 나노초 |
| 8 | Order ref number | 주문 고유 ID |
| 1 | Buy/Sell | 매수/매도 구분 |
| 4 | Shares | 주식 수량 |
| 8 | Stock symbol | 종목 심볼 |
| 4 | Price | 가격 |

### 오더북에 필요한 메시지 종류
| 분류 | 타입 | 값 |
|------|------|-----|
| Add | Add order | `A = 0x41` |
| Add | Add w/ MPID | `F = 0x46` |
| Cancel | Order executed | `E = 0x45` |
| Cancel | Executed w/ price | `C = 0x43` |
| Cancel | Order cancel | `X = 0x58` |
| Cancel | Order delete | `D = 0x44` |
| Cancel | Order replace | `U = 0x55` |

> 파서의 역할: 위 메시지만 골라 오더북으로 넘기고 나머지는 버린다. `replace`는 사실상 cancel+add 두 개로 분해 가능.
> **재현 팁**: 실제 ITCH 스펙(`NQTV-ITCH-V4_1`)을 받아두면 정확한 필드 오프셋을 확인할 수 있다. 처음엔 add/cancel 두 종류만 지원하고 점차 확장하는 게 현실적이다.

## 3. 오더북 (Order Book) — 가장 까다로운 자료구조

### 개념
시장의 현재 상태를 압축해 표현한 자료구조. **양쪽(bid/ask)** 을 가지며, 각 가격대(price ladder)별로 도착 순서대로 정렬된 주문 리스트를 유지한다. "같은 가격이면 먼저 온 주문 우선" 규칙.

- 개념적으로는 **가격을 키로 하는 이진탐색트리 + 가격별 시간순 연결리스트**
- **depth(깊이)**: 추적하는 서로 다른 가격의 개수
- 상태는 BRAM에 저장

### 지원 연산 3가지
| 연산 | 입력 |
|------|------|
| `AddOrder` | stock_symbol, order_id, price, quantity |
| `CancelOrder` | order_id |
| `getBestPrice` | stock_symbol → best_buy, best_sell |

### 구현 난이도 — 두 가지 접근

**(A) 기본 구현 — BRAM 배열 스캔**
- 주문들을 BRAM에 배열로 저장
- **Add**: 끝에 추가 → 6사이클 (상수, 빠름)
- **Delete**: order_id를 찾으려 BRAM을 처음부터 스캔 → 책 크기에 비례. 50개 주문이면 **244사이클**
- 문제: delete 지연이 비결정적 → HFT엔 부적합

**(B) 트리 기반 구현 (스트레치 목표) — 핵심 아이디어**

관찰: 주식 가격은 하루 중 좁은 범위에서 움직이고, ticker(호가 단위, 예: 0.1) 배수로만 존재한다. 그래서:

1. **가격 → 인덱스 매핑**: `(price − X) / ticker`로 가격을 인덱스로 변환
2. **가격 레지스터 뱅크(P_Q)**: 각 가격의 누적 수량 저장
3. **리듀서 트리**: 레지스터 위에 트리를 쌓아 최우선 호가를 계산. 노드는 "오른쪽 자식 수량이 0이 아니면 그 값, 아니면 왼쪽 자식 값"을 취함 (최대가격 추적)
4. order_id로 인덱싱하는 BRAM은 별도 유지

```
Level 0 (root) ─ 최우선 호가
   ▲
Level 1  [노드][노드]
   ▲
Level 2  [노드][노드][노드][노드]
   ▲
Level 3 (leaves) ─ 각 가격의 누적 수량  ← P_Q 레지스터 뱅크
```

- 트리 높이 = log₂n (n = 서로 다른 가격 수)
- **Add/Delete 모두 7사이클** (원본 측정, 예측 높이 6에 근접) → 기본 구현 대비 엄청난 속도 향상이자 **결정적** 지연
- **주의**: 트리를 조합논리(combinational)로 연결하면 타이밍 위반 → 클럭 단계(pipeline)로 나눠야 함. 그러면 갱신 지연이 log₂n 사이클로 늘지만 throughput은 유지

> **재현 시 교훈 (원작자 Endrias)**:
> - C 자료구조를 그대로 포팅하지 말 것. FPGA의 병렬성에 맞는 설계를 다시 생각하라 ("Think FPGA").
> - 같은 BRAM에 대한 접근을 관리하는 **메모리 매니저 모듈**을 따로 두면 코드가 훨씬 단순해진다 (포트당 1연산 제약 준수).
> - Vivado 경고를 무시하지 말 것.

## 4. 트레이딩 로직 — 최소분산 포트폴리오

### 전략 개념
시장 위험을 최소화하는 **최소분산 포트폴리오(minimum variance portfolio)** 를 유지한다. 핵심은 종목 수익률의 **공분산 행렬 K**.

목표 비중 w:
```
        K⁻¹ · 1
w = ───────────────
     (K⁻¹ · 1)ᵀ · 1
```

> 행렬을 **직접 역행렬할 필요는 없다.** 선형 시스템 `K·v = 1`을 풀고 정규화하면 됨.

### 공분산 행렬을 매 시점 갱신하기

수익률 관측: 현재가 p, 1ms 전 가격 q일 때 `r = (q − p)/p − 1`. 이렇게 얻은 수익률 벡터를 x라 하자.

공분산 정의: `K = E[XXᵀ] − E[X]E[X]ᵀ`

새 관측 x_{N+1}이 오면 두 항을 **증분 갱신**:
```
              N · Eᴺ[XXᵀ] + x_{N+1}·x_{N+1}ᵀ
Eᴺ⁺¹[XXᵀ] = ──────────────────────────────────
                       N + 1

           N              x_{N+1}
Eᴺ⁺¹[X] = ───── · Eᴺ[X] + ───────
          N+1               N+1
```
→ 매 단계 `Eᴺ[XXᵀ]`(2차 모멘트 행렬)와 `Eᴺ[X]`(평균)만 저장하면 됨.

**실시간 Risk 계산**: 위험은 `wᵀKw`. 매번 전체 행렬-벡터 곱을 하지 않고, 새 관측의 기여분 `wᵀvvᵀw = 2(vᵀw)` 만 증분 더하면 됨 → **단순 내적**으로 환원 → 가산기 트리로 수십 사이클(나노초 단위)에 계산. HEX 디스플레이로 표시.

### 선형 시스템 풀기 — QR 분해 + CORDIC

`K·v = 1`을 풀 때 고정소수점 오차에 강하도록 **QR 분해** 후 상삼각 시스템을 푼다.

```
K = QR  (Q: 직교행렬, R: 상삼각행렬)
K·v = 1  →  R·v = Q⁻¹·1
```

**QR 분해 방법 3가지와 선택**
- Householder / Gram-Schmidt: 병렬 곱셈기가 많은 FPGA에 적합 → Artix-7엔 DSP가 부족(240개)해서 부적합
- **Givens 회전 (채택)**: 행렬의 한 원소씩 0으로 만들어 상삼각화. 두 행을 회전시킴. DSP 사용을 최소화 → Artix-7에 적합

**CORDIC(코딕)**: 곱셈기 없이 **시프트와 덧셈만으로** 삼각함수/회전을 계산하는 알고리즘. Givens 회전에 필요한 arctan 계산과 벡터 회전을 CORDIC 블록 2개(arctan용 + rotator용)로 처리.
- N×N 행렬엔 N(N−1)/2번의 Givens 회전 필요
- 서로 다른 행을 쓰는 회전들은 **병렬**로 묶어 stage로 나눔
- folded 설계(같은 CORDIC 블록 재사용)로 면적 절약

### 지연 분석 (4×4 기준, 원본 측정)
| 단계 | 사이클 |
|------|--------|
| 공분산 갱신 | ~70 |
| QR 분해 (CORDIC) — **병목** | ~210 |
| 상삼각 풀이 (back-substitution) | ~90 |
| 정규화 + 주문 생성 | ~50 |
| **합계** | **~340–400** (80MHz에서 ~425μs) |

- CPU(Intel Xeon, numpy) 대비 약 2배 빠르고 **결정적**
- CORDIC 반복 횟수(정밀도)를 낮추면 더 빨라지지만 오차↑ (14회→5회 시 MSE 3.9%→13%)

### 설계 철학: Folded vs Pipelined
- **Folded**: 회로 하나를 여러 stage가 재사용 → 면적 작음
- **Pipelined**: stage마다 전용 회로 → throughput 높음(매 사이클 새 입력)
- 단일 입력 지연은 동일. Nexys 4의 자원 제약 때문에 **folded 채택**.

## 5. FPGA 하드웨어 핵심 개념

### 주요 자원
| 자원 | 설명 | 이 프로젝트에서 |
|------|------|----------------|
| **LUT** | 조합논리 구현 | 트레이딩 로직 11.0K(18%) |
| **FF (플립플롭)** | 레지스터/상태 저장 | 14.1K(11%) |
| **BRAM** | 블록 메모리 | 오더북 저장 (20만 주문 시 13.7%) |
| **DSP** | 곱셈/MAC 전용 블록 | 트레이딩 52개(20%), Artix-7 총 240개 |

### Ethernet 스택 — MicroBlaze
- **MicroBlaze**: Xilinx의 소프트코어 프로세서(FPGA 패브릭 위에 구현하는 CPU)
- TCP/IP 스택(lwIP)을 C로 돌려 네트워크 통신 처리
- FPGA 로직과는 **AXI-Stream** 프로토콜로 데이터 교환
- ⚠️ **원본에서 통합 실패한 지점**: MicroBlaze의 FSL 명령어 불안정, ILA/SDK 셋업 문서 부족 등. 데모는 결국 Ethernet 없이 FPGA 내부에서 직접 주문을 생성해 우회함.
- 대안(스트레치): MicroBlaze 대신 경량 Ethernet 스택을 쓰면 패킷→파서 파이프라인이 더 타이트해짐

### AXI / AXI-Stream
- Xilinx IP들이 데이터를 주고받는 표준 인터페이스
- `tvalid`/`tready` 핸드셰이크, `tuser`로 파이프라인 토큰 추적
- ⚠️ Block Design에서 AXI 포트의 reset은 **active-low**여야 함 (원본 gotcha)

### 고정소수점 (Fixed Point) — "Fixed Point Hell"
- FPGA엔 부동소수점이 비싸서 **고정소수점**(예: FP(2,14) = 정수 2비트 + 소수 14비트)을 씀
- ⚠️ 정수부/소수부 너비를 **반드시 파라미터화**하라. 스케일링 팩터를 곳곳에서 추적해야 함
- 곱셈 후 스케일 보정(`/ 2^14`), 오버플로 처리에 주의
- Xilinx divider IP: 정수부·소수부 **둘 다 부호 비트**가 있음 (매뉴얼 정독 필수)

### Latency-insensitive 설계 (모듈 간 동기화)
모듈마다 끝나는 시점이 다르므로:
- **reset-ready 프로토콜**: 시작 전 reset → 끝나면 `ready` high. 파이프라인 입력이 없을 때 사용 (이 프로젝트 주력)
- **tuser 토큰 프로토콜**: 파이프라인 입력마다 고유 토큰을 붙여 출력에서 매칭 (AXI-Stream 방식)

## 6. 실제 보드 재현 — 단계별 권장 순서

각 모듈은 **독립적으로 시뮬레이션 검증 → FPGA 합성 → 통합** 순으로 진행하는 게 원본의 흐름이다. 그리고 원작자들이 가장 후회한 부분이 **"통합을 너무 늦게 시작한 것"** 이니, 통합을 일찍·자주 하는 걸 권장한다.

1. **준비**: 보드 선정(Nexys 4 / Artix-7 계열), Vivado 설치, 클럭/리셋/디스플레이 같은 기본 인프라 확인
2. **각 모듈을 Python/C++ 레퍼런스로 먼저 작성** → 나중에 RTL 검증용 테스트벤치의 정답지로 사용 (원본이 실제로 쓴 방법)
3. **파서**: ITCH 바이트 FSM부터. UART로 입력받아 파싱 결과 확인 → 후에 AXI-Stream으로 전환
4. **오더북**: 소프트웨어 구현 → 기본(BRAM 배열) RTL → 트리 기반 최적화
5. **트레이딩 로직**: 공분산 갱신 → CORDIC QR 솔버 → 정규화/주문 생성
6. **통합**: 오더북+트레이딩 먼저, 그다음 Ethernet (가장 어려움 — 일찍 손대기)
7. **검증**: 각 단계마다 Python 출력과 RTL 시뮬레이션 결과를 cross-check, 보드에서는 ILA로 신호 관찰
